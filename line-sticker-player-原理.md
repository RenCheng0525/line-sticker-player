# LINE Sticker Player — 運作原理小報告

## 一、概觀

LINE Sticker Player 是一支**純前端、單一 HTML 檔**的小工具,沒有後端、沒有資料庫,所有邏輯都在瀏覽器裡執行。它把對 LINE 貼圖的三件事整合在兩個分頁中:**瀏覽**(從商店找出有哪些貼圖組)、**播放**(把一組貼圖的動畫放出來看)、以及**保存**(整組打包下載,或產生可離線使用的版本)。

整個工具能成立,靠的是三個觀察:LINE 的貼圖資源放在規則固定的公開 CDN 上、動畫貼圖用的是瀏覽器原生支援的 APNG 格式、以及「顯示圖片」和「讀取位元組」在瀏覽器的同源政策下待遇不同。以下逐一說明。

## 二、貼圖資源從哪裡來

LINE 的貼圖檔案存放在公開內容傳遞網路 `stickershop.line-scdn.net`,每種資源都有固定的網址規則。只要知道一組貼圖的 **product id** 或單張的 **sticker id**,就能組出對應的網址:

| 用途 | 網址樣式 |
| --- | --- |
| 清單(metadata) | `/product/{pid}/iphone/productInfo.meta` — 一段 JSON,內含所有 `stickers[].id` |
| 靜態圖(首格) | `/sticker/{id}/iphone/sticker@2x.png` |
| 動畫檔(APNG) | `/sticker/{id}/iphone/sticker_animation@2x.png` |
| 全螢幕 / 彈出 | `/sticker/{id}/iphone/sticker_popup@2x.png` |
| 商店縮圖 | `/product/{id}/LINEStorePC/main.png` |
| 官方整包 ZIP | `/product/{pid}/iphone/stickerpack@2x.zip` |

這些端點是非官方、未公開的慣例,LINE 隨時可能調整。

## 三、為什麼「播放」幾乎不費力:APNG

動畫貼圖的格式是 **APNG(Animated PNG)**——由多張帶序號的影格組成一個會動的 PNG。關鍵在於現代瀏覽器(Chrome、Safari、Firefox)**原生支援** APNG:只要把動畫檔的網址放進一個 `<img>` 標籤,它就會自己循環播放,不需要任何解碼器、函式庫或 canvas 繪圖。

換句話說,播放器的核心其實只有一句話:**把正確的網址放進 `<img>`**。工具大部分的程式碼不是在「播放」,而是在「想辦法拿到那串網址」。

## 四、三層架構與資料流

工具的運作可以拆成一條由上而下的資料流:

1. **瀏覽層**:抓取商店某個分類頁的內容 → 從中解析出每一組貼圖的 product id、名稱、類型 → 用每組的 `main.png` 當縮圖,排成清單。
2. **播放層**:拿到一個 product id → 抓 `productInfo.meta` → 取出該組所有 sticker id → 每個 id 各自組出動畫網址 → 放進 `<img>` 播放。
3. **保存層**:把每張動畫的位元組抓下來 → 用 JSZip 打包成 ZIP,或轉成內嵌資料生成一份離線 HTML。

使用者在「瀏覽」點某組的「開啟」,等於是把該組的 product id 交給播放層;載入後再交給保存層處理。

## 五、最大的障礙:同源政策(CORS),以及解法

這是整個工具設計上最關鍵的一點。瀏覽器對跨網站的存取有兩種截然不同的待遇:

- **顯示圖片(`<img src>`)不受 CORS 限制。** 所以只要拿得到 sticker id,動畫**一定播得出來**,這條路永遠暢通。
- **用 JavaScript 跨網站讀取內容(`fetch`)會被 CORS 擋住。** 抓商店頁面、抓 `productInfo.meta`、以及抓圖片的「位元組」拿去打包,這些都需要對方伺服器回傳 CORS 授權標頭,而 LINE 的網站並不提供。

解法是兩道保險。第一道是**多代理輪詢**:抓取時依序嘗試多個公開 CORS 代理(`r.jina.ai`、allorigins、corsproxy、thingproxy),任何一個成功就採用;因為不同代理回傳的格式不一(有的回 HTML、jina 回 Markdown),解析器設計成兩種都能吃。第二道是**完全手動的保底**:就算所有代理都失效,使用者仍可自行在瀏覽器打開頁面、複製內容貼進工具(貼清單、貼 id、或貼 meta JSON),這條路完全不連外,保證可用。

## 六、播放 / 暫停是怎麼做到的

APNG 放在 `<img>` 裡會自動播放,但**沒有原生的暫停**。工具用兩種情境、兩種招式解決:

- **線上播放器**:暫停 = 把 `src` 換成該張的靜態首格(`sticker@2x.png`);播放 = 換回動畫檔(並加上一個變動參數讓動畫從頭重播)。
- **離線版**:因為離線版的圖是內嵌的 data URI(屬同源、不會污染 canvas),暫停時就把當下那一格畫到一個 `<canvas>` 上定格、蓋在動畫之上;放開時再把 `<img>` 顯示回來。

## 七、離線版的原理

「產生離線版」會把每張 APNG 抓成位元組 → 轉成 base64 的 data URI → 連同必要的 CSS 與控制用 JavaScript 一起,**內嵌**寫進一份全新的 HTML。產物是**單一檔、零外部相依**:之後在任何地方打開,完全不需要連網就能瀏覽與播放/暫停。代價是檔案較大(所有圖都包在裡面),這是用體積換取「離線、可攜」的取捨。

## 八、支援範圍與限制

LINE 的特殊貼圖分好幾類,工具對它們的支援程度不同:

- **動態 / 有聲動態**:動畫正常播放(聲音目前未播放)。
- **靜態 / 大貼圖**:沒有動畫檔,自動退回顯示靜態圖。
- **全螢幕 / 特效**:它們的動畫是另一種 `popup` 類資源,目前播放器只嘗試一般動畫網址,因此會退回靜態(可再擴充支援)。
- **訊息 / 隨你填**:這兩類的重點是使用者輸入、即時合成的文字,單純抓圖無法重現。

此外要留意:公開代理本來就時靈時不靈、有流量限制;CDN 端點屬非官方慣例,隨時可能變動;貼圖版權屬原作者所有,本工具僅供個人預覽研究,不應散布或商用。

## 九、技術小結

整體是一支不依賴後端的前端工具,核心相依只有四項:瀏覽器**原生 APNG** 播放、**JSZip** 負責打包、**公開 CORS 代理**負責繞過跨來源限制、以及 **FileReader + Blob** 負責生成離線檔。設計上的中心思想,是把「一定可行的事」(用 `<img>` 顯示)和「可能被擋的事」(用 `fetch` 讀取)清楚分開,並為後者層層準備備援,讓工具在網路條件不理想時仍有路可走。
